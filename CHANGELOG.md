## [Unreleased]

### Исправлено
- BLAST Bounty матчи реально появились на Polymarket (24.07) — и сразу
  вскрыли три реальных расхождения с тем, что мы предполагали при
  подключении (проверено живым запросом, не угадано):
  - **Тег `--matches` был не тот**: `cs2` тихо возвращал 0 матчей всё это
    время (без ошибки!) — реальный тег матчевых событий BLAST —
    `counter-strike-2` (подтверждено по фактическим тегам живого
    события: `[esports, counter-strike-2, games, sports]`).
  - **`title_filter` был не тот**: реальные заголовки матчей — `"Counter-
    Strike: Sharks vs FURIA (BO3) - BLAST Bounty Qualifier"`, не
    содержат "2026 Season 2". Новый фильтр — `"BLAST Bounty"` (с
    заглавных, без "Qualifier") — специально выбран так, чтобы НЕ
    матчить prop-эвенты (те пишутся `"Blast Bounty 2026 Season 2: ..."`,
    другой регистр — подтверждено отсутствие ложных срабатываний).
    Пометка на будущее: если у этапа 2 (QF/SF/GF) заголовок будет другим
    — фильтр придётся поправить ещё раз, по факту, не заранее.
  - **`--matches-since` фильтрует по дате СОЗДАНИЯ события на Gamma, не
    по дате самого матча** — матчи с реальными датами 21-24.07 были
    заведены на Gamma 17.07, поэтому граница `2026-07-20` тихо (без
    ошибки) отсекала их все. Сдвинута на `2026-07-15`.
  - **`collect._is_result_market` не узнавал CS2-терминологию карт**:
    Dota пишет `"Game N Winner"`, CS2 — `"Map N Winner"` (подтверждено
    на реальном событии FURIA-Sharks) — резолвы по отдельным картам CS2
    тихо не собирались (только общий "Match Winner" резолвился
    нормально, т.к. это название одинаковое у обеих игр).
    `RESULT_MARKET_PREFIX` → `RESULT_MARKET_PREFIXES = ("Game ", "Map ")`.
  - Живой прогон после фикса: 48 рынков увидено (16 матчей × 3 типа),
    32 leg-снапшота, 32 новых авто-прогноза (12 с моделью, 20 market-only).

### Добавлено
- `evhedge scenarios --db --tournament --ranks [--teams] [--flag] [--emit-configs]`:
  сценарное дерево фиксированной сетки этапа 2 (8 команд, QF→SF→GF,
  без пересева) — как только известна восьмёрка.
  - `enumerate_stage2_scenarios`: полный перебор 2^7=128 исходов
    (проверено: сумма вероятностей = 1.0; при всех парных p=0.5 каждая
    из 8 команд даёт ровно 12.5% — точь-в-точь санити из промпта).
    Силы команд — из свежих book-verified winner-мидов в БД
    (`auto_predict.latest_book_winner_mid`), n на стадию — из `--ranks`
    (8 посеянных команд должны иметь ОДИНАКОВЫЙ n на входе; SF/GF
    получаются арифметически n−1/n−2, а не из отдельного конфига).
  - `team_outlooks`: P(титул) и P(выигрыш стадии | дошла) на команду —
    последнее усреднено по всем возможным соперникам ветки (соперник на
    SF/GF заранее не известен).
  - `stage_conflict`: карта конфликтов детерминирована ЧИСТО по слотам
    сетки (не вероятностная) — на какой стадии два флагнутых участника
    могут встретиться; на очный матч ноги сознательно не предлагаются.
  - `--emit-configs`: черновики `examples/<team>_stage2_scenario.yaml`
    в формате `evhedge ev` (team/stages/market/strategy) —
    win_prob по стадиям помечены `MODEL-ESTIMATE (power_model_v1)` в
    комментарии, `market.no_price` берётся из реального book-ask
    `winner_no`, если он есть, иначе честно помечаемый комплемент
    модели. Проверено живым прогоном: сгенерированный файл грузится
    через `load_full_config` без правок.
  - `detect_qf_pairs`: best-effort автоопределение 4 QF-пар из
    `ps_results` (нужно ровно 4 матча со стадией "QF"/"Quarterfinal");
    при неоднозначности — явная ошибка с указанием задать `--teams`
    вручную, никогда не угадывает.
- `evhedge stageranks suggest --db --tournament --stage-map --ranks
  [--apply]`: полуавтоматический пересчёт `stage_ranks` из PandaScore-
  зеркала (`ps_results`). Карта «стадия → n» — отдельный конфиг
  (`configs/blast_bounty_s2_stage_map.yaml` для BLAST: Ro32→5, Ro16→4,
  QF/Quarterfinal→3, SF/Semifinal→2, Final/Grand Final→1), сопоставление
  по подстроке названия стадии PandaScore (регистронезависимо, самая
  длинная подстрока побеждает — иначе "Final" ложно match'ился бы внутри
  "Semifinal").
  - `compute_stage_ranks`: n победившей команды считается АРИФМЕТИЧЕСКИ
    (n_предыдущей_стадии − 1), а не поиском следующей стадии в
    PandaScore — команде не нужно ждать, пока там заведут матч
    следующего раунда, чтобы её n обновился. Выбывшие — исключены из
    карты полностью (честный `NULL` у автопредикта при отсутствии
    записи, не догадка). Чемпион (n дошёл бы до 0) — тоже исключён,
    ставить больше не на что.
  - Автоправки без подтверждения ЗАПРЕЩЕНЫ: без `--apply` команда
    только показывает дифф (добавить/изменить/убрать), никогда не
    трогает файл — прогнозы неизменяемы, устаревший n дешевле
    ошибочного (та же дисциплина, что и у ручного обновления карты).
- PandaScore (бесплатный Fixtures-тариф) — независимый от Polymarket
  источник расписаний/результатов: `data_sources/pandascore.py`,
  таблица `ps_results` (схема v7→v8), `evhedge pandascore sync`,
  `evhedge reconcile`, `evhedge deadlines`.
  - Клиент: только list-эндпоинты (`/csgo/matches/{upcoming,past,running}`,
    `/csgo/series`, `/csgo/teams`) — все подтверждены "All plans" по

    используется. Пагинация — по `Link: rel="next"` (тот же паттерн, что

  - Шаг 0 живым запросом (2026-07-20): `BLAST Bounty 2026 Season 2` в
    PandaScore ЕЩЁ НЕ ЗАВЕДЕНА (ни серии, ни турнира, ни единого матча
    под лигой BLAST Premier, `league_id=4321`) — турнир стартует 24.07,
    та же временная дыра, что и с Gamma match-событиями. Реестр команд

    есть отдельная несвязанная команда "Falcons", id 131216 — разные

    (`team_aliases.yaml`), новых записей ради PandaScore заводить не
    пришлось.
  - `pandascore_sync.py`: `match_to_ps_result` канонизирует команды ЧЕРЕЗ
    ТУ ЖЕ `team_aliases.canonical_name`, что и `collect.py` — не второй
    матчер. `ps_results` — НЕ пишется в `resolves` (DESIGN CHOICE,
    зафиксирован в докстринге `PSResult`): resolves остаётся правдой

  - `evhedge deadlines --hours N` (по умолчанию 3ч): ближайшие матчи из

  - `blast_watch.bat`: добавлен шаг `pandascore sync` (по

### Удалено
- `evhedge webapp`: блок "Стоимость единицы хеджа / Объём хеджирования /
  Расходы на хедж" убран целиком — не соответствовал реальной стратегии.

  вычитания посторонней суммы.

### Исправлено
- `evhedge webapp`: "Загрузить" на реальной позиции подставляло фиктивный
  коэффициент 0.85/"Завершено" из дефолтной пустой строки, никак не
  учитывая ФАКТИЧЕСКИЙ исход позиции — на реальном примере пользователя
  (Team Yandex проиграл, currentValue=0) это рисовало вымышленную
  прибыль +$555.92 вместо настоящего убытка -100%. Теперь при
  `redeemable=true` коэффициент считается из
  `(currentValue-initialValue)/initialValue`, статус — "Завершено"; при
  ещё не резолвнутом рынке — коэффициент сбрасывается в 0, статус — "В
  ожидании" (реального исхода ещё нет, дефолтную заглушку не показываем
  как посчитанный результат).

  `predictions`, как только у leg-рынка ("Match Winner") появляется
  качественная книга — убирает ручной шаг калибровочного контура и лечит
  два измеренных дефекта: протухшие модельные входы (прогноз, зафиксированный
  вручную ПОСЛЕ того, как доска уже ушла) и селекционный биас (человек
  предсказывает то, что заметил, а не каждый рынок с книгой).
  - Триггер (`book_quality_trigger`): первый снапшот, где книга реальная
    (`source="book"`), обе стороны присутствуют, спред ≤5пп (константа
    `BOOK_QUALITY_SPREAD_MAX_PP`). Обоснование — реальный кейс из БД:
    листинг BetBoom–Poor Rangers 4.0/96.0 вечером, живая цена только к

    отдельный обход Gamma) — переиспользует уже сделанный вызов

    канонические имена). Запрет power_model.py на неоднородные n
    применён на уровне ПАРЫ (n_a == n_b), а не всего турнира — проверено
    фактически на `examples/wc2026_bracket.yaml`: 4 пары 1/8 финала имеют
    n=4 каждая, при этом уже прошедшая в 1/4 Brazil имеет n=3 — то есть

    `...|p_model=NULL` без модели) — версия модели (`MODEL_VERSION =
    "power_model_v1"`) обязательна в обеих формах: без неё калибровочный
    ряд нельзя честно сравнивать после будущего изменения γ/формулы.

    структура описывает целое дерево сетки одной команды, тяжелее, чем
    нужно); отдельный плоский YAML, который пользователь вручную двигает
    по ходу турнира, путь передаётся явно (`pull --stage-ranks`), без
    дефолтного файла.

    авто-записано (с моделью / market-only), и — что оказалось более
    полезной метрикой, чем "ждёт триггера/скипнуто" из исходного

  когда ноги закотируют. Тесты: fuel-фикстуры на числах реальных находок

  класс сигнала. Каждый результат несёт обязательное поле
  `caveat = VERIFY_BOOK_CAVEAT` («verify book before trading») — caveat
  является частью данных, а не документацией, по PROJECT RULE из
  `data_sources/polymarket.py`. Три проверки:
  - `basket_check(markets, slots)` — корзина NO-асков доски с
    фиксированным числом слотов против гарантированной выплаты
    `(n - slots) * 100` (находка «+1.2% корзина»);
  - `identity_check(parent_market, member_markets)` — агрегатный рынок
    должен стоить как сумма взаимоисключающих членов, маппинг членов —
    руками в конфиге, автовывода нет намеренно (находка «CONCACAF=USA
    +0.6%»); DESIGN CHOICE: `IDENTITY_MIN_EDGE_PCT = 0.5` — зазор ниже
    полупункта не отличим от несвежей витрины и спреда;
  - `vertical_check(team, ladder)` — цепочка reach_X цен команды
    монотонна по глубине, условные `p_cond` правдоподобны: `p_cond >= 1`
    — жёсткое нарушение (сигнал), экстремум — мягкий флаг; DESIGN CHOICE:
    границы экстремумов `VERTICAL_EXTREME_LOW/HIGH = 0.05/0.95`.
  `tests/test_consistency.py` — фикстуры воспроизводят числа реальных
  находок (+1.2%, +0.6%).

### Изменено
- `StrategyConfig.kelly_fraction`: дефолт изменён с `1.0` (full Kelly) на
  `0.5` (half Kelly) — вероятности `p_r` являются собственной оценкой
  модели, а не рыночной котировкой, и full Kelly чрезмерно чувствителен к
  ошибке этой оценки. Риск full Kelly теперь нужно запрашивать явно
  (`kelly_fraction=1.0`).
- Границы ответственности между `MarketPrices` и `StrategyConfig`
  пересмотрены дважды:
  1. Изначально `no_stake_usd`/`bankroll` были частью `MarketPrices`.
  2. Затем перенесены в `StrategyConfig`, а `MarketPrices` сведён только к
     сырым рыночным ценам (`no_price`, `yes_price`) — чтобы в будущем
     `data_sources/polymarket.py` мог наполнять `MarketPrices` напрямую с
     биржевого API, не смешивая рыночные данные с решением о размере
     позиции.
- `compute_hedge_plan` соответственно менял сигнатуру: сначала
  `(stages, strategy)`, затем `(stages, strategy, market)` (когда
  `no_stake_usd`/`bankroll` временно жили в `MarketPrices`), и в итоге
  вернулся к `(stages, strategy)` после переноса этих полей обратно в
  `StrategyConfig`. Позже `market` добавлен обратно в четвёртый раз — на
  этот раз осознанно и навсегда: режиму `hedge_mode="lock_in"` реально
  нужна `market.no_price` для расчёта `net_no_win` на первом шаге, а не
  по ошибке, как в предыдущих итерациях.
